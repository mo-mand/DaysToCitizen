# From EC2 to AWS Amplify — a 250-word migration recap

DaysToCitizen — the Canadian-citizenship day tracker at `daystocitizen.ca` — started its life on a single AWS EC2 `t3.micro` in Ohio. Nginx, PM2, a Let's Encrypt cert, and a flat JSON file on disk. It worked, it was easy to debug, and it cost about $5.70/month once the Free Tier ran out. What it wasn't was fast for users outside North America, and the operational overhead — security patches, certificate renewals, manual backups — was growing instead of shrinking.

So I rewrote the cloud architecture from scratch and moved everything to serverless on AWS Amplify. The Next.js app now runs on Amplify's `WEB_COMPUTE` platform: CloudFront serves static assets from global edge locations, and SSR routes fall through to a managed Lambda in `ca-central-1`. The flat-file database became three DynamoDB tables on PAY_PER_REQUEST billing. Secrets Manager holds the Resend API key and JWT signing secret. Every piece of the stack — IAM roles, tables, secrets, the Amplify app, the domain association — is defined in Terraform, so the whole architecture can be reproduced or torn down with one command.

The migration surfaced seven distinct bugs, most memorable being a trust policy that needed *both* `amplify.amazonaws.com` and `lambda.amazonaws.com` as principals before the SSR Lambda would accept AWS credentials at runtime. After fixing them, PageSpeed Insights went from 87 to 90, Speed Index dropped 40%, Total Blocking Time hit zero, and the monthly bill collapsed from ~$5.70 to effectively zero. The full write-up is in [MigrationFromEC2toAmplify.md](./MigrationFromEC2toAmplify.md).
